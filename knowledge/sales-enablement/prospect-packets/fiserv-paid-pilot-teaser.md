# Fiserv Aevah Paid Pilot Teaser

## Internal Objective

Prepare a high-credibility teaser for an onsite Fiserv meeting. The conversation should position Aevah as a practical way to prove data unification and workflow acceleration in Fiserv's own environment while preserving operating continuity across existing systems.

## Recommended Opening Angle

Fiserv's portal and customer workflow challenges appear to come from acquired-company fragmentation: multiple systems, multiple data models, multiple workflows, and too much human coordination. Aevah can be positioned as a governed operating layer that unifies one high-value workflow first, proves value, then expands across groups.

## Pilot Hypothesis

Deploy Aevah inside Fiserv's environment for one specific customer workflow or data domain. The pilot should prove:

- Data can be unified across selected source systems.
- Customer workflow time can be reduced.
- Business users can interact with trusted operational data.
- Lineage, security, and governance can be preserved.
- Expansion opportunities can be mapped by group or budget owner.

## Candidate Pilot Lanes

### Customer Portal Workflow Acceleration

- Map the current customer submission workflow.
- Identify duplicate handoffs, data re-entry, and portal fragmentation.
- Create a unified operating view for the workflow.
- Measure time saved from submission to completion.

### Acquired-System Data Product

- Select one data domain from acquired-company systems.
- Define source systems, entity model, data quality rules, and governance.
- Build a reusable governed data product.
- Demonstrate how the product supports portal UX or process automation.

### Entity And Golden Record Foundation

- Identify customer, product, card, or account entities that create workflow friction.
- Create matching, survivorship, and lineage rules for a focused domain.
- Show how consistent entity data improves customer workflow speed.

## Offer Fit

- Aevah: primary platform motion.
- Aevah MDM Golden Record And Entity Resolution: strong fit if entity consistency is the first blocker.
- Aevah Automated Migration Tool: possible fit if legacy mappings, business rules, or acquired-system migration are in scope.
- Product Development And Intelligent Systems: possible support for portal workflow and integration buildout.

## Meeting Ask

Ask for an onsite working session in Alpharetta with Eric, Adam, and any data, portal, architecture, security, and customer workflow stakeholders. The session should identify one pilot lane, one data domain, and one success metric.

## Commercial Framing

Use paid pilot language only after the stakeholder confirms scope and appetite. Internally discussed ranges should not be sent externally until approved.

Possible structure:

- Fixed-scope paid pilot.
- Deployed into Fiserv-owned environment.
- Clear success metrics.
- Expansion roadmap after pilot.
- Optional license and support model after pilot.

## Human Review Needed

- Confirm stakeholder names, titles, and meeting attendees.
- Confirm current cloud/security/deployment constraints.
- Confirm whether prior project history can be referenced.
- Leadership review required before discussing license, perpetual license, support fee, or pilot pricing.
